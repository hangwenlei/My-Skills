# 设计文档：给 `sync` 插件加「去重 / 智能合并」能力

- 日期：2026-06-10
- 状态：已确认，待制定实现计划
- 仓库：https://github.com/hangwenlei/My-Skills.git（增强既有 `sync` 插件，命令 `/sync:docs`）
- 关联文件：`plugins/sync/skills/docs/SKILL.md`

## 1. 背景与目标

现有 `/sync:docs` 的步骤 4「刷新其它文档」只判断文档是否**过时**，**不识别跨文档的内容冗余**。实际项目里同一事实常在多个文档重复（本仓库就是活样本：`sync 是什么 / 怎么装 / 为什么带命名空间` 这套事实在 README、CLAUDE.md、HANDOFF.md、`docs/superpowers/specs` 各讲了一遍）。同一件事改一处就要同步改好几个文档，维护成本高、易不一致。

目标：让 `/sync:docs` 在刷新文档时**顺带识别冗余并建议收敛/合并**，遵循主流 DRY / 单一信息源（SSOT）思想；但要「看情况」——日志/时间线类 append-only 文档不能合并；并且整件事必须**轻量**，不能让 `/sync:docs` 变成一次「大手术」。

## 2. 关键决策

| 决策点 | 结论 | 理由 / 被否方案 |
|--------|------|----------------|
| 去重层面 | **跨文档收敛/合并**为主，HANDOFF 自身去重为附带 | 用户真实痛点是「同一件事多个文档各维护一份」。 |
| 能不能合并 | **先给文档分类**：日志/时间线型一律跳过，仅叙述/状态型进入去重候选 | 日志的顺序与历史本身是信息，合并会丢失语义。 |
| 扫描范围 | **只比对本次改动牵连到的主题**所涉及的文档，不做全项目两两比对 | 「不做大手术」的关键闸门；全项目 SSOT 重构（方案 C）正是被否的「大手术」。 |
| 自动程度 | **标记 + 确认后改**：列建议清单，用户勾选后才改，改完给 diff | 否决「超集自动合并」（方案 B，更重、易丢边角、易揉错受众）。与现有步骤 4 的 propose-confirm 完全一致，零学习成本。 |
| 不丢信息 | 收敛/合并前**逐条确认被收敛方独有信息已在保留方出现**，做不到就不动 | 合并的红线是不丢信息。 |
| 受众边界 | 受众不同的文档（README 面向用户 vs spec 面向开发者）**默认不物理合并**，只建议加指针 | 保住各文档独立可读性，避免揉成一团。 |
| 交互与提交 | 仍 propose-confirm、**不自动 commit** | 沿用现状，提交权留给用户。 |

### 被否方案
- **方案 B（超集自动合并）**：检测到「文档2 = 文档1 + 增量」就直接生成合并草案替换。更接近「智能合并」但更重、风险更高——已将其「超集即合并」能力降级为清单里的一种**动作类型**（走确认而非自动）吸收进推荐方案。
- **方案 C（全项目 SSOT 重构）**：一次扫全项目彻底收敛。这正是用户明确排斥的「大手术」，排除。

## 3. 设计：对 `SKILL.md` 的改动

改动**只落在步骤 4（扩展）+ 步骤 2（一句附带增强）**，不新增步骤、不改步骤 1/3/5，保持轻量。

### 3.1 步骤 4 扩展为「刷新过时 + 收敛冗余」

在现有步骤 4 内部插入以下逻辑（仍是「列清单 → 等确认 → 只改确认项」的总框架）：

**4.0 文档分类（前置判断，决定能不能动）**

扫描候选文档时先分两类：
- **日志型 / 时间线型 → 一律跳过，不参与去重**。识别启发式（命中任一即判定）：
  - 文件名或一级标题含 `log` / `changelog` / `journal` / `日志` / `时间线` / `记录` / `ADR` / `decision`；
  - **或** 正文主体是带日期戳、按时间排列的条目列表。
- **叙述型 / 状态型**（README、设计文档、说明等）→ 进入去重候选。

**4.1 聚焦范围**

只比对「本次 session 改动（步骤 1）+ 对话上下文实际涉及的主题」所牵连到的那几个叙述型文档，**不做全项目两两比对**。

**4.2 识别两类问题，合并进同一张建议清单**

- **过时**（原有能力）：文档内容与本次改动不符。
- **冗余 / 可合并**（新增）：
  - `可收敛`：两个及以上叙述型文档在讲同一事实 → 建议**留权威出处**、其它处改成一句话 + 指针/链接。
  - `可合并`：文档2 基本是文档1 的超集 → 建议合并成一份，保留全部独有信息。

**4.3 清单每项标注**（在现有「路径 + 为什么 + 拟改要点」基础上加 `类型`）：
- 文档路径
- **类型**：`过时` / `可收敛` / `可合并`
- 为什么
- 拟改要点：收敛/合并时指明**保留方**、被收敛方改成什么、并声明「已核对独有信息不丢」

**4.4 护栏**（执行确认项时遵守）：
- **不丢信息**：执行收敛/合并前逐条确认被收敛方的独有信息已在保留方出现；做不到就不动该项，只在清单里点出供用户决定。
- **受众边界**：受众不同的文档默认不物理合并，只建议加指针。
- **权威出处规则**：面向用户的事实 → README 为权威；架构/设计决策 → spec/设计文档为权威；当前开发现场 → HANDOFF 为权威；项目级长期约束 → CLAUDE.md 为权威。

**4.5 不变项**：仍停下等用户勾选；仅改确认项；不 commit。

### 3.2 步骤 2 附带增强（HANDOFF 自身去重）

在步骤 2「快照式整体重写」的说明里补一句：**同一事实只保留一条，不在多个分节里重复出现**。零成本，符合 DRY。

## 4. 最终效果

跑 `/sync:docs` 时，用户会在原有「建议更新清单」里多看到带 `可收敛` / `可合并` 标记的条目，并附「日志型已跳过」的说明。勾哪个改哪个，不勾则与现状完全一致。不会变重、不会自动删内容、不会揉错受众。

## 5. 测试与验证

沿用现有「临时 git 项目手动按 SKILL.md 步骤实测」的方式，新增针对去重的用例：

1. **可合并（超集）**：建两个叙述型文档，文档2 = 文档1 + 一段独有内容。运行 → 清单出现 `可合并` 条目，拟改要点指明保留方且声明不丢独有内容；确认后合并结果包含两者全部独有信息。
2. **可收敛（重复事实）**：让 README 与某说明文档讲同一事实。运行 → 出现 `可收敛` 条目，建议留权威出处、别处改指针。
3. **日志型跳过**：建一个 `CHANGELOG.md` / 含日期条目的日志文档，且内容与别处重复。运行 → 该文档**不**出现在去重清单（被分类跳过）。
4. **受众边界**：README（面向用户）与 spec（面向开发者）内容重叠。运行 → 默认建议加指针而非物理合并。
5. **不确认即不改**：对去重条目全不勾选 → 相关文档零改动。
6. **HANDOFF 自身**：步骤 2 重写后，同一事实不在多个分节重复。

## 6. 非目标（YAGNI）

- 不做全项目两两比对 / 全量 SSOT 重构（大手术）。
- 不做超集自动合并（一律走确认）。
- 不物理合并受众不同的文档（只加指针）。
- 不碰日志/时间线型文档。
- 不自动 commit。
- 不改未经确认的文档。
